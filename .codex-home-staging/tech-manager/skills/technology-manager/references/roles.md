# Role Briefs

## frontend-developer

Use for:

- UI implementation
- component behavior
- visual polish
- client-side state
- accessibility and responsive adjustments

Delegate with:

- exact feature or screen scope
- owned files or modules
- behavior and acceptance criteria
- constraints from the existing design system or product language

Expected return:

- changed files
- user-visible behavior
- known limitations
- follow-up verification notes

## backend-developer

Use for:

- API routes
- service logic
- database access
- schema changes
- integrations

Delegate with:

- exact endpoint, service, or workflow scope
- owned files or modules
- data contracts and failure cases
- required validation or migration concerns

Expected return:

- changed files
- contract or behavior changes
- migration or env impacts
- known risks

## quality-assurance

Use for:

- acceptance validation
- regression review
- test plan creation
- gap identification
- release readiness checks

Delegate with:

- the feature or bug being verified
- expected acceptance criteria
- the environments or commands available
- known weak points to target

Expected return:

- pass or fail status
- concrete findings with severity
- reproduction steps
- coverage or testing gaps

## business-analyst

Use for:

- clarifying ambiguous requirements
- identifying hidden assumptions
- defining scope boundaries
- translating product intent into acceptance criteria

Delegate with:

- the raw request or notes
- business context if known
- conflicting interpretations that need sorting

Expected return:

- clarified problem statement
- assumptions
- open questions
- crisp acceptance criteria

## project-manager

Use for:

- milestone planning
- work breakdown
- dependency mapping
- delivery sequencing
- blocker tracking

Delegate with:

- desired outcome
- delivery window if known
- active roles or teams
- dependencies and constraints

Expected return:

- phased plan
- owners
- dependency notes
- critical path and blockers

## performance-engineer

Use for:

- profiling slow flows
- runtime bottleneck analysis
- query or API latency review
- render and bundle performance
- caching and throughput improvements

Delegate with:

- the slow path or target metric
- the environment or commands available for measurement
- budgets or thresholds if known
- known hotspots or suspected regressions

Expected return:

- bottlenecks found
- recommended fixes
- measured or estimated impact
- verification approach

## security-engineer

Use for:

- authentication and authorization review
- secret handling
- input validation and abuse prevention
- dependency and supply-chain risk review
- secure defaults for deployment and data handling

Delegate with:

- the feature, flow, or deployment surface under review
- trust boundaries and sensitive data if known
- applicable constraints such as auth model or compliance needs
- known concerns or suspected weak points

Expected return:

- findings with severity
- concrete fixes or hardening steps
- residual risk
- verification approach

## reliability-engineer

Use for:

- deployment readiness
- uptime and failure-mode review
- monitoring and observability gaps
- rollback and recovery planning
- production risk reduction

Delegate with:

- the service, flow, or release under review
- runtime and deployment environment if known
- availability goals or operational constraints
- known incidents or brittle areas

Expected return:

- reliability risks
- recommended safeguards
- rollout or rollback notes
- verification and monitoring approach
